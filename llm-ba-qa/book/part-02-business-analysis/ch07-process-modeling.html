7.1 Understanding Business Processes

A business process is a sequence of activities that transforms inputs into outputs to deliver value to a customer or stakeholder. Processes range from simple (expense approval) to complex (end-to-end order fulfilment spanning multiple departments and systems).

Process Hierarchy

Level Name Description Example
L0 Value Chain End-to-end enterprise capabilities Order to Cash
L1 Process Group Major functional processes Sales, Fulfilment, Billing
L2 Process Defined sequence of activities Order Processing
L3 Sub-Process Detailed steps within a process Credit Check, Inventory Allocation
L4 Task Atomic unit of work Verify customer address

BPMN 2.0 Essentials

BPMN (Business Process Model and Notation) is the industry standard for visually representing business processes. The core elements are:

  • Events: Start (circle), Intermediate (double circle), End (thick circle) — represent triggers and outcomes.
  • Activities: Tasks (rounded rectangles) and Sub-Processes (rounded rectangles with a "+" marker).
  • Gateways: Exclusive (X), Parallel (+), Inclusive (O) — control flow branching and merging.
  • Flows: Sequence flows (solid arrows), Message flows (dashed arrows), Associations (dotted lines).
  • Swimlanes: Pools and Lanes that represent participants and departments.
Why BPMN Matters: BPMN is not just documentation. Modern BPM engines (Camunda, jBPM, Flowable) can directly execute BPMN XML definitions. An LLM that generates valid BPMN produces artifacts that are both human-readable and machine-executable.

7.2 Process Discovery from Documents

Process discovery is the act of understanding how work actually gets done. Traditional approaches involve interviews, workshops, and process mining from system logs. LLMs add a new capability: extracting process descriptions from the text-heavy documents that already exist in every organisation — SOPs, training manuals, email threads, and audit reports.

Process Extraction Prompt


import openai, json, textwrap

PROCESS_DISCOVERY_PROMPT = textwrap.dedent("""\
    You are a business process analyst. Read the following document
    and extract all business processes described or implied.

    For each process, provide:
      - process_id: PROC-001, PROC-002, ...
      - name: descriptive process name
      - trigger: what initiates the process
      - actors: list of roles/departments involved
      - steps: ordered list of activities, each with:
          - step_number: sequential integer
          - actor: who performs this step
          - action: what is done
          - inputs: what is needed
          - outputs: what is produced
          - decision_point: true/false (is this a gateway?)
          - conditions: if decision_point is true, the branching conditions
      - end_state: what constitutes process completion
      - exceptions: known exception paths
      - estimated_duration: if mentioned or inferable

    DOCUMENT:
    {document_text}

    Return a JSON array of process objects.
""")


def discover_processes(document_text: str) -> list[dict]:
    """Extract business process descriptions from unstructured text."""
    response = openai.chat.completions.create(
        model="gpt-4o",
        temperature=0.1,
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": "You are a business process analyst."},
            {"role": "user", "content": PROCESS_DISCOVERY_PROMPT.format(
                document_text=document_text
            )},
        ],
    )
    result = json.loads(response.choices[0].message.content)
    if isinstance(result, dict):
        result = next(iter(result.values()))
    return result

Validating Discovered Processes

Extracted processes should be validated against these quality checks:

  • Completeness: Does every process have a clear start trigger and end state?
  • Consistency: Are actor names consistent across processes? (e.g., "Finance Dept" vs. "Accounting Team")
  • Granularity: Are all processes at roughly the same level of detail?
  • Coverage: Do the extracted processes cover the entire scope of the source document?

def validate_processes(processes: list[dict]) -> dict:
    """Run quality checks on discovered processes."""
    issues = []
    all_actors = set()

    for proc in processes:
        # Check completeness
        if not proc.get("trigger"):
            issues.append({"process": proc["process_id"], "issue": "Missing trigger"})
        if not proc.get("end_state"):
            issues.append({"process": proc["process_id"], "issue": "Missing end state"})
        if len(proc.get("steps", [])) < 2:
            issues.append({"process": proc["process_id"], "issue": "Too few steps (< 2)"})

        # Collect actors for consistency check
        for step in proc.get("steps", []):
            all_actors.add(step.get("actor", "Unknown"))

    # Check for actor name inconsistencies (simple similarity check)
    actor_list = sorted(all_actors)
    for i, a1 in enumerate(actor_list):
        for a2 in actor_list[i+1:]:
            if a1.lower().replace(" ", "") == a2.lower().replace(" ", "") and a1 != a2:
                issues.append({
                    "process": "ALL",
                    "issue": f"Inconsistent actor names: '{a1}' vs '{a2}'"
                })

    return {
        "total_processes": len(processes),
        "total_steps": sum(len(p.get("steps", [])) for p in processes),
        "unique_actors": len(all_actors),
        "issues": issues,
        "quality_score": max(0, 100 - len(issues) * 10),
    }
Tip: Run process discovery on multiple documents from the same department, then ask the LLM to reconcile the results. Different documents often describe the same process at different levels of detail or with different terminology. The reconciliation step produces a more complete picture than any single source.

7.3 BPMN Generation with LLMs

Once a process is described in structured JSON, the next step is generating a formal BPMN 2.0 XML definition that can be imported into modelling tools (Camunda Modeler, Signavio, Bizagi) or executed by a BPM engine.

BPMN XML Generation


BPMN_PROMPT = textwrap.dedent("""\
    Convert the following process description into valid BPMN 2.0 XML.

    RULES:
    1. Use proper BPMN 2.0 namespace declarations.
    2. Include start event, end event, and all intermediate steps.
    3. Use exclusive gateways for decision points.
    4. Use parallel gateways where steps can occur simultaneously.
    5. Assign each task to the correct lane within a pool.
    6. Include sequence flows connecting all elements.
    7. The XML must be valid and importable into Camunda Modeler.

    PROCESS:
    {process_json}

    Return only the XML. No markdown code fences.
""")


def generate_bpmn_xml(process: dict) -> str:
    """Generate BPMN 2.0 XML from a structured process description."""
    response = openai.chat.completions.create(
        model="gpt-4o",
        temperature=0.0,
        messages=[
            {"role": "system", "content": "You generate valid BPMN 2.0 XML."},
            {"role": "user", "content": BPMN_PROMPT.format(
                process_json=json.dumps(process, indent=2)
            )},
        ],
    )
    xml_content = response.choices[0].message.content
    # Strip any accidental markdown fences
    if xml_content.startswith("```"):
        xml_content = "\n".join(xml_content.split("\n")[1:-1])
    return xml_content

Validating the Generated BPMN


from lxml import etree

BPMN_SCHEMA_URL = "https://www.omg.org/spec/BPMN/20100501/BPMN20.xsd"

def validate_bpmn(xml_string: str) -> dict:
    """Validate BPMN XML against the BPMN 2.0 schema."""
    try:
        doc = etree.fromstring(xml_string.encode("utf-8"))

        # Basic structural checks (schema validation requires the XSD file)
        namespaces = {"bpmn": "http://www.omg.org/spec/BPMN/20100524/MODEL"}
        start_events = doc.findall(".//bpmn:startEvent", namespaces)
        end_events = doc.findall(".//bpmn:endEvent", namespaces)
        tasks = doc.findall(".//bpmn:task", namespaces) + doc.findall(".//bpmn:userTask", namespaces)
        gateways = (
            doc.findall(".//bpmn:exclusiveGateway", namespaces)
            + doc.findall(".//bpmn:parallelGateway", namespaces)
        )
        sequence_flows = doc.findall(".//bpmn:sequenceFlow", namespaces)

        return {
            "valid_xml": True,
            "start_events": len(start_events),
            "end_events": len(end_events),
            "tasks": len(tasks),
            "gateways": len(gateways),
            "sequence_flows": len(sequence_flows),
            "errors": [],
        }
    except etree.XMLSyntaxError as e:
        return {"valid_xml": False, "errors": [str(e)]}
Warning: LLM-generated BPMN XML often has minor issues: missing sequence flow connections, incorrect namespace prefixes, or overlapping diagram coordinates. Always validate the XML programmatically and visually inspect it in a modelling tool before sharing with stakeholders.

7.4 Bottleneck Identification

A bottleneck is any step in a process that limits the throughput of the entire system. Traditional bottleneck analysis uses process mining on event logs. LLMs complement this by analysing qualitative data — interview transcripts, complaint tickets, and delay reports — to identify bottlenecks that may not appear in structured logs.

Quantitative Bottleneck Analysis


import pandas as pd

def analyse_process_metrics(event_log: pd.DataFrame) -> pd.DataFrame:
    """Identify bottlenecks from process event log data.

    Expected columns: case_id, activity, start_time, end_time, resource
    """
    # Calculate duration for each activity instance
    event_log["duration_hours"] = (
        (event_log["end_time"] - event_log["start_time"]).dt.total_seconds() / 3600
    )

    # Aggregate metrics per activity
    metrics = event_log.groupby("activity").agg(
        avg_duration=("duration_hours", "mean"),
        median_duration=("duration_hours", "median"),
        p95_duration=("duration_hours", lambda x: x.quantile(0.95)),
        case_count=("case_id", "nunique"),
        resource_count=("resource", "nunique"),
    ).reset_index()

    # Calculate utilisation and wait ratios
    metrics["throughput_per_resource"] = metrics["case_count"] / metrics["resource_count"]
    metrics["variability"] = metrics["p95_duration"] / metrics["median_duration"]

    # Flag bottlenecks: high duration + high variability + low resource count
    metrics["bottleneck_score"] = (
        metrics["avg_duration"].rank(pct=True) * 0.4
        + metrics["variability"].rank(pct=True) * 0.3
        + (1 - metrics["resource_count"].rank(pct=True)) * 0.3
    )

    return metrics.sort_values("bottleneck_score", ascending=False)

Qualitative Bottleneck Analysis with LLMs


BOTTLENECK_PROMPT = textwrap.dedent("""\

""")


def identify_bottlenecks(

        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": "You are a process improvement specialist."},
            {"role": "user", "content": BOTTLENECK_PROMPT.format(
                process_json=json.dumps(process, indent=2),
                metrics_json=json.dumps(metrics, indent=2),
                feedback_text=feedback or "No qualitative feedback available.",
            )},
        ],
    )
    result = json.loads(response.choices[0].message.content)
    if isinstance(result, dict):
        result = next(iter(result.values()))

Key Insight: The most valuable bottleneck insights often come from combining quantitative and qualitative data. A step with average metrics might still be a bottleneck if interview data reveals it causes frustration, errors, or workarounds that are not captured in the event log.

7.5 Process Optimization Suggestions

Once bottlenecks are identified, the next step is generating improvement recommendations. LLMs can propose optimisations based on established process improvement methodologies: Lean (eliminate waste), Six Sigma (reduce variation), and automation-first strategies.

Optimization Recommendation Engine


OPTIMIZATION_PROMPT = textwrap.dedent("""\
    You are a process improvement consultant. Given the process
    description and identified bottlenecks, propose specific,
    actionable optimisations.

    For each recommendation, provide:
      - opt_id: OPT-001, OPT-002, ...
      - target_activity: the step being improved
      - category: ELIMINATE | AUTOMATE | SIMPLIFY | PARALLELIZE | OUTSOURCE | STANDARDIZE
      - description: detailed description of the change
      - expected_impact: quantified improvement estimate
      - implementation_effort: LOW | MEDIUM | HIGH
      - risk: what could go wrong
      - prerequisites: what must be in place first
      - roi_estimate: qualitative ROI assessment

    PROCESS:
    {process_json}

    BOTTLENECKS:
    {bottlenecks_json}

    Prioritise recommendations by impact-to-effort ratio.
    Return a JSON array.
""")


def generate_optimizations(process: dict, bottlenecks: list[dict]) -> list[dict]:
    """Generate prioritised process optimization recommendations."""
    response = openai.chat.completions.create(
        model="gpt-4o",
        temperature=0.3,
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": "You are a Lean Six Sigma Black Belt consultant."},
            {"role": "user", "content": OPTIMIZATION_PROMPT.format(
                process_json=json.dumps(process, indent=2),
                bottlenecks_json=json.dumps(bottlenecks, indent=2),
            )},
        ],
    )
    result = json.loads(response.choices[0].message.content)
    if isinstance(result, dict):
        result = next(iter(result.values()))
    return result

Optimization Categories Explained

Category Description When to Apply Example
Eliminate Remove the step entirely Step adds no value (pure waste) Remove redundant approval for orders under $100
Automate Replace manual work with software Step is rule-based and repetitive Auto-generate invoices from order data
Simplify Reduce complexity of the step Step has unnecessary sub-steps or forms Replace 12-field form with 4-field smart form
Parallelise Run steps concurrently Steps are independent Run credit check and inventory check simultaneously
Outsource Delegate to a specialist team or service Step requires expertise not core to the business Use a third-party identity verification service
Standardise Create a consistent procedure Step is done differently by different people Create a standard checklist for quality review
Tip: Present optimisations on a 2x2 impact-effort matrix. High-impact, low-effort items are "quick wins" that build momentum. Schedule them first to demonstrate value before tackling larger changes.

7.6 As-Is to To-Be Mapping

As-Is/To-Be mapping is the standard BA technique for visualising the current state of a process alongside the proposed future state. LLMs can generate the To-Be model by applying the optimisation recommendations to the As-Is process.

Generating the To-Be Process


TO_BE_PROMPT = textwrap.dedent("""\
    You are redesigning a business process. Given the current (As-Is)
    process and a set of approved optimisations, produce the future
    (To-Be) process.

    OUTPUT FORMAT:
    Return a JSON object with:
      - to_be_process: same structure as the As-Is process but reflecting
        all applied optimisations
      - changes_applied: list of changes, each with:
          - opt_id: the optimisation that drove this change
          - as_is_step: the original step(s) affected
          - to_be_step: the new step(s) (or "REMOVED" if eliminated)
          - change_type: ADDED | MODIFIED | REMOVED | REORDERED
      - metrics_comparison: estimated before/after for:
          - total_steps
          - estimated_cycle_time
          - manual_steps_count
          - decision_points

    AS-IS PROCESS:
    {as_is_json}

    APPROVED OPTIMISATIONS:
    {optimizations_json}
""")


def generate_to_be_process(as_is: dict, optimizations: list[dict]) -> dict:
    """Generate a To-Be process by applying optimisations to the As-Is model."""
    response = openai.chat.completions.create(
        model="gpt-4o",
        temperature=0.2,
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": "You are a process redesign specialist."},
            {"role": "user", "content": TO_BE_PROMPT.format(
                as_is_json=json.dumps(as_is, indent=2),
                optimizations_json=json.dumps(optimizations, indent=2),
            )},
        ],
    )
    return json.loads(response.choices[0].message.content)

Comparison Report


def format_comparison_report(mapping_result: dict) -> str:
    """Format an As-Is vs To-Be comparison report."""
    metrics = mapping_result.get("metrics_comparison", {})
    changes = mapping_result.get("changes_applied", [])

    lines = [
        "# As-Is vs To-Be Process Comparison",
        "",
        "## Metrics Summary",
        f"| Metric | As-Is | To-Be | Change |",
        f"|--------|-------|-------|--------|",
    ]

    for key, label in [
        ("total_steps", "Total Steps"),
        ("estimated_cycle_time", "Cycle Time"),
        ("manual_steps_count", "Manual Steps"),
        ("decision_points", "Decision Points"),
    ]:
        before = metrics.get(f"{key}_before", metrics.get(key, {}).get("before", "N/A"))
        after = metrics.get(f"{key}_after", metrics.get(key, {}).get("after", "N/A"))
        lines.append(f"| {label} | {before} | {after} | -- |")

    lines.extend(["", "## Changes Applied", ""])
    for change in changes:
        lines.append(f"- **{change['change_type']}**: {change.get('as_is_step', 'N/A')} -> {change.get('to_be_step', 'N/A')} (from {change.get('opt_id', 'N/A')})")

    return "\n".join(lines)
Presentation Tip: When presenting As-Is/To-Be to executives, use a side-by-side visual with the As-Is on the left and To-Be on the right. Highlight removed steps in red, new steps in green, and modified steps in amber. The metrics comparison table at the bottom provides the business case.

7.7 Change Impact Analysis

Before implementing process changes, you must understand the ripple effects. A change impact analysis identifies which systems, roles, policies, and training materials are affected by a process modification.

Impact Analysis Framework


IMPACT_PROMPT = textwrap.dedent("""\
    Perform a change impact analysis for the following process changes.

    For each change, assess impact across these dimensions:
      - people: roles affected, training needs, headcount implications
      - process: upstream/downstream processes affected
      - technology: systems, integrations, and data flows impacted
      - policy: regulations, SOPs, or governance documents that need updating
      - risk: new risks introduced by the change

    For each impacted item, provide:
      - dimension: PEOPLE | PROCESS | TECHNOLOGY | POLICY | RISK
      - item: what is affected
      - impact_level: HIGH | MEDIUM | LOW
      - description: how it is affected
      - action_required: what must be done to address the impact

    CURRENT PROCESS:
    {as_is_json}

    PROPOSED CHANGES:
    {changes_json}

    ORGANISATIONAL CONTEXT:
    {org_context}

    Return a JSON object with: total_impacts (int), high_impacts (int),
    impacts (array), readiness_score (0-100), go_no_go_recommendation (string).
""")


def change_impact_analysis(
    as_is: dict,
    changes: list[dict],
    org_context: str = "",
) -> dict:
    """Perform change impact analysis across people, process, technology, policy, and risk."""
    response = openai.chat.completions.create(
        model="gpt-4o",
        temperature=0.2,
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": "You are an organisational change management specialist."},
            {"role": "user", "content": IMPACT_PROMPT.format(
                as_is_json=json.dumps(as_is, indent=2),
                changes_json=json.dumps(changes, indent=2),
                org_context=org_context or "No additional context provided.",
            )},
        ],
    )
    return json.loads(response.choices[0].message.content)

Visualise impact levels across dimensions and changes using a heatmap:


def print_impact_heatmap(impacts: list[dict]) -> None:
    """Print a text-based impact heatmap."""
    dimensions = ["PEOPLE", "PROCESS", "TECHNOLOGY", "POLICY", "RISK"]
    level_symbol = {"HIGH": "!!!", "MEDIUM": " ! ", "LOW": " . "}

    # Group by dimension
    by_dimension = {d: [] for d in dimensions}
    for impact in impacts:
        dim = impact.get("dimension", "UNKNOWN")
        if dim in by_dimension:
            by_dimension[dim].append(impact)

    print(f"\n{'Dimension':<14} {'Count':>5}  {'High':>4}  {'Med':>4}  {'Low':>4}")
    print("-" * 45)
    for dim in dimensions:
        items = by_dimension[dim]
        high = sum(1 for i in items if i["impact_level"] == "HIGH")
        med  = sum(1 for i in items if i["impact_level"] == "MEDIUM")
        low  = sum(1 for i in items if i["impact_level"] == "LOW")
        print(f"{dim:<14} {len(items):>5}  {high:>4}  {med:>4}  {low:>4}")
Warning: Change impact analysis is where LLMs are most likely to miss organisation-specific factors — union agreements, legacy system constraints, cultural resistance patterns. Always supplement LLM output with input from change management practitioners who know the organisation.

Build an end-to-end tool that takes operational documents as input and produces a complete process analysis package: discovered processes, BPMN diagrams, bottleneck analysis, optimisation recommendations, and a To-Be process model with change impact assessment.

Main Pipeline


# process_engine/main.py
import json, pathlib, argparse


def main():
    parser = argparse.ArgumentParser(description="Process Discovery Engine")
    parser.add_argument("input_dir", type=pathlib.Path, help="Directory of source documents")
    parser.add_argument("--output", type=pathlib.Path, default=pathlib.Path("output"))
    parser.add_argument("--metrics", type=pathlib.Path, help="Optional event log CSV")
    args = parser.parse_args()

    args.output.mkdir(exist_ok=True)

    # Step 1: Discover processes from all documents
    all_processes = []
    for doc_file in sorted(args.input_dir.glob("*.txt")):
        text = doc_file.read_text(encoding="utf-8")
        print(f"[Discovery] Processing {doc_file.name}...")
        processes = discover_processes(text)
        all_processes.extend(processes)

    print(f"[Discovery] Found {len(all_processes)} processes across all documents")

    # Step 2: Validate
    validation = validate_processes(all_processes)
    print(f"[Validation] Quality score: {validation['quality_score']}/100")

    # Step 3: Generate diagrams for each process
    for proc in all_processes:
        mermaid = generate_mermaid(proc)
        filename = proc["process_id"].lower().replace("-", "_")
        (args.output / f"{filename}_diagram.md").write_text(mermaid, encoding="utf-8")

    # Step 4: Bottleneck analysis
    for proc in all_processes:
        bottlenecks = identify_bottlenecks(proc, [], "")
        proc["bottlenecks"] = bottlenecks

        # Step 5: Optimisations
        if bottlenecks:
            optimizations = generate_optimizations(proc, bottlenecks)
            proc["optimizations"] = optimizations

            # Step 6: To-Be mapping
            to_be = generate_to_be_process(proc, optimizations)
            proc["to_be"] = to_be

            # Step 7: Change impact
            changes = to_be.get("changes_applied", [])
            impact = change_impact_analysis(proc, changes)
            proc["change_impact"] = impact

    # Write final output
    (args.output / "full_analysis.json").write_text(
        json.dumps(all_processes, indent=2), encoding="utf-8"
    )
    print(f"[Complete] Full analysis written to {args.output}/full_analysis.json")


if __name__ == "__main__":
    main()

Summary

  • Process discovery from documents allows LLMs to extract structured process descriptions from SOPs, manuals, and transcripts — providing a starting point that would take days to produce manually.
  • BPMN generation produces machine-readable process models, though validation and visual inspection are essential before use.
  • Bottleneck identification benefits most from combining quantitative metrics with qualitative LLM analysis of interviews and complaints.
  • Six optimisation categories (Eliminate, Automate, Simplify, Parallelise, Outsource, Standardise) provide a systematic framework for improvement recommendations.
  • As-Is/To-Be mapping with LLMs produces a clear before-and-after comparison with estimated metrics improvements.
  • Change impact analysis across people, process, technology, policy, and risk dimensions ensures that process changes are implemented with full awareness of their ripple effects.

Exercises

Conceptual

  1. Explain why an LLM might discover a process step that does not appear in any single document but is implied by the combination of multiple documents. Give an example.
  2. Compare the strengths of process mining (from event logs) versus LLM-based process discovery (from documents). When would you use each, and when would you combine them?
  3. A stakeholder objects to the To-Be process because it eliminates their role as an approver. How would you handle this using the change impact analysis framework?

Coding

  1. Write a function that detects circular dependencies in a set of processes (Process A feeds into Process B, which feeds into Process A).
  2. Extend the validate_bpmn function to check that every task has at least one incoming and one outgoing sequence flow (no orphaned tasks).
  3. Build a function that generates a process simulation — given a To-Be process and assumed step durations, calculate the expected end-to-end cycle time for 100 simulated cases.

Design

  1. Design a "Process Health Dashboard" that monitors live processes and uses LLM analysis to flag emerging bottlenecks before they become critical. Sketch the architecture and data flows.
  2. Propose an integration between the Process Discovery Engine and a BPM execution platform (e.g., Camunda). How would you ensure the generated BPMN is deployment-ready?
