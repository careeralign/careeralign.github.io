Part 3 — Quality Assurance with LLMs

Defect Analysis and Triage

Every mature software team drowns in defects. Hundreds of bugs sit in the backlog, many are duplicates, severity labels are inconsistent, and triage meetings burn hours debating priorities. In this chapter, you will build an LLM-powered defect triage assistant that classifies, deduplicates, and prioritizes defects — turning a chaotic backlog into an actionable plan.

Reading time: ~25 min Project: Defect Triage Assistant

What You Will Learn

  • How to classify defects automatically by type, component, and root cause using LLMs
  • Techniques for detecting duplicate and near-duplicate defect reports
  • Performing root cause analysis by correlating defect descriptions with code changes
  • Building a severity and priority scoring model that reduces triage time
  • Recognizing defect patterns that indicate systemic quality issues
  • Assessing regression risk when prioritizing fixes

1. The Defect Flood

A typical enterprise project accumulates defects at an alarming rate. Consider these industry benchmarks:

MetricSmall Team (5 devs)Medium Team (20 devs)Large Team (100+ devs)
New defects per sprint10–2040–80200–500
Backlog size (open defects)50–100200–5001,000–5,000+
Duplicate rate10–15%15–25%25–40%
Time spent in triage meetings1 hr/week3–5 hrs/week10–20 hrs/week
Average time to triage a defect5 min5 min5 min
Total triage cost per sprint1.5 hrs6 hrs40+ hrs

The cost is not just time — it is decision quality. When a QA lead triages 50 defects in a one-hour meeting, each defect gets barely a minute of attention. Severity is assigned inconsistently. Duplicates slip through. High-impact bugs get buried under a pile of cosmetic issues.

LLMs can process the full text of a defect report — title, description, steps to reproduce, stack traces, screenshots-to-text — and make classification decisions in seconds. This does not eliminate the need for human judgment, but it provides a strong first pass that dramatically reduces the time and cognitive load of triage.

The real cost of bad triage. Research by Capers Jones shows that a defect found in production costs 10-100x more to fix than one found during testing. But a defect mislabeled as low priority during triage and left to fester until it hits production costs even more — because the team had the information to fix it early and failed to act. Better triage is not just about efficiency; it is about preventing production incidents.

2. Automated Defect Classification

The first step in taming the defect flood is consistent classification. LLMs can categorize defects along multiple dimensions simultaneously:

from openai import OpenAI
import json

client = OpenAI()

CLASSIFICATION_PROMPT = """You are a senior QA engineer classifying defects.
Analyze the defect report and classify it along these dimensions:

1. DEFECT TYPE: functional | performance | security | usability | data |
   integration | configuration | documentation
2. COMPONENT: identify the most likely system component/module affected
3. ROOT CAUSE CATEGORY: code_logic | data_handling | concurrency |
   configuration | third_party | requirements_gap | environment
4. AFFECTED USER SEGMENT: all_users | specific_role | specific_browser |
   specific_region | specific_data_pattern
5. REPRODUCIBILITY: always | intermittent | rare | environment_specific

Also extract:
- key_symptoms: array of observable symptoms mentioned in the report
- related_features: array of features likely affected
- suggested_assignee_team: which team should investigate

Return JSON with all classification fields."""

def classify_defect(defect: dict) -> dict:
    """Classify a defect report along multiple dimensions."""
    defect_text = f"""Title: {defect.get('title', '')}
Description: {defect.get('description', '')}
Steps to Reproduce: {defect.get('steps', '')}
Expected Result: {defect.get('expected', '')}
Actual Result: {defect.get('actual', '')}
Environment: {defect.get('environment', '')}
Stack Trace: {defect.get('stack_trace', '')}
Reporter: {defect.get('reporter', '')}
Date: {defect.get('date', '')}"""

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": CLASSIFICATION_PROMPT},
            {"role": "user", "content": defect_text}
        ],
        response_format={"type": "json_object"},
        temperature=0.2
    )
    classification = json.loads(response.choices[0].message.content)
    return {**defect, "classification": classification}


# Example defect
defect = {
    "id": "BUG-4521",
    "title": "Payment fails intermittently for amounts over $10,000",
    "description": """Users report that payments over $10,000 sometimes fail
    with a generic error message. The issue seems to happen more frequently
    on weekends. The payment gateway returns a timeout error after 30 seconds.
    Smaller payments process normally.""",
    "steps": """1. Log in as any user
    2. Initiate a payment of $15,000
    3. Select wire transfer as payment method
    4. Confirm payment
    5. Observe: ~30% of the time, payment fails after 30-second delay""",
    "expected": "Payment processes successfully within 5 seconds",
    "actual": "Payment times out after 30 seconds with error 'Transaction failed'",
    "environment": "Production, all browsers",
    "stack_trace": "TimeoutException at PaymentGateway.processTransaction()",
    "reporter": "Sarah Chen (Customer Support)",
    "date": "2025-11-15"

classified = classify_defect(defect)
c = classified["classification"]
print(f"Defect: {defect['title']}")
print(f"  Type: {c['defect_type']}")
print(f"  Component: {c['component']}")

The classification output provides structured metadata that would take a human 3-5 minutes to determine:

Defect: Payment fails intermittently for amounts over $10,000
  Type: integration
  Component: payment_gateway
  Root cause: third_party (gateway timeout under high-value transactions)
  Reproducibility: intermittent

Build a classification feedback loop. Track cases where the LLM's classification is overridden during triage. Feed these corrections back as few-shot examples in the prompt. After 50-100 corrections, classification accuracy typically improves from 70-75% (out of the box) to 85-90% (fine-tuned with your team's patterns).

3. Root Cause Analysis with LLMs

Beyond classification, LLMs can perform preliminary root cause analysis by correlating the defect description with recent code changes, system architecture, and known issues. This transforms triage from "what happened?" to "why did it happen?" — before a developer even looks at the bug.

RCA_PROMPT = """You are a senior software engineer performing root cause
analysis. Given a defect report and recent code changes, determine:

1. PROBABLE ROOT CAUSE: What most likely caused this defect? Be specific.
2. CONTRIBUTING FACTORS: What other factors may have contributed?
3. EVIDENCE: What evidence from the defect report supports your analysis?
4. INVESTIGATION STEPS: What should the developer check first?
5. SIMILAR PAST ISSUES: What patterns suggest this is a recurring problem?
6. CONFIDENCE: How confident are you in this analysis? (high/medium/low)

Return JSON with these keys."""

def analyze_root_cause(defect: dict,
                       recent_changes: list[dict] = None,
                       known_issues: list[dict] = None) -> dict:
    """Perform preliminary root cause analysis on a defect."""
    context = f"""Defect Report:
Title: {defect['title']}
Description: {defect['description']}
Steps: {defect.get('steps', 'Not provided')}
Stack Trace: {defect.get('stack_trace', 'Not provided')}
Environment: {defect.get('environment', 'Not provided')}
"""

    if recent_changes:
        context += f"\nRecent Code Changes:\n{json.dumps(recent_changes, indent=2)}"
    if known_issues:
        context += f"\nKnown Issues:\n{json.dumps(known_issues, indent=2)}"

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": RCA_PROMPT},
            {"role": "user", "content": context}
        ],
        response_format={"type": "json_object"},
        temperature=0.3
    )
    return json.loads(response.choices[0].message.content)

Root cause analysis is hypothesis generation, not proof. The LLM's analysis is a starting point for investigation, not a conclusion. Always verify the hypothesis with actual log analysis, code review, and debugging. The value is in giving the developer a focused starting point rather than a blank slate.

4. Duplicate Detection

Duplicate defects waste everyone's time. The reporter spends time filing a bug that already exists. The triager spends time reading and classifying it. Sometimes both the original and the duplicate get assigned to different developers, who waste time investigating the same issue. Industry data suggests that 15–40% of defect reports are duplicates.

Traditional duplicate detection uses keyword matching and fails on bugs described in different words. LLMs understand semantics — they recognize that "login button unresponsive" and "cannot click sign-in after page loads" describe the same issue.

DUPLICATE_PROMPT = """You are a defect deduplication specialist.
Compare the NEW defect against each EXISTING defect and determine
if the new defect is a duplicate, related, or unique.

For each comparison, provide:
- existing_id: the ID of the existing defect
- relationship: "duplicate" | "related" | "unique"
- confidence: 0.0 to 1.0
- reasoning: why you think they are related or not
- key_similarities: what they share
- key_differences: what distinguishes them

A defect is a DUPLICATE if it describes the same underlying issue,
even if the symptoms, steps, or wording are different.
A defect is RELATED if it involves the same component or feature
but describes a different issue.

Return JSON with key "comparisons" sorted by confidence (highest first),
and a "verdict" field with the overall recommendation."""

def detect_duplicates(new_defect: dict,
                      existing_defects: list[dict],
                      threshold: float = 0.7) -> dict:
    """Check if a new defect is a duplicate of any existing defect."""
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": DUPLICATE_PROMPT},
            {"role": "user", "content": f"""NEW DEFECT:
{json.dumps(new_defect, indent=2)}

EXISTING DEFECTS:
{json.dumps(existing_defects, indent=2)}"""}
        ],
        response_format={"type": "json_object"},
        temperature=0.2
    )
    result = json.loads(response.choices[0].message.content)

    # Filter to matches above threshold
    result["high_confidence_matches"] = [
        c for c in result["comparisons"]
        if c["confidence"] >= threshold
    ]
    return result

Batch Deduplication

For cleaning an entire backlog, process defects in batches and build a cluster map:

def deduplicate_backlog(defects: list[dict]) -> dict:
    """Identify all duplicate clusters in a defect backlog."""
    clusters = {}  # canonical_id -> [duplicate_ids]
    processed = set()

    for i, defect in enumerate(defects):
        if defect["id"] in processed:
            continue

        # Compare against all remaining unprocessed defects
        remaining = [d for d in defects[i+1:] if d["id"] not in processed]
        if not remaining:
            continue

        result = detect_duplicates(defect, remaining, threshold=0.75)
        duplicates = [
            m["existing_id"] for m in result.get("high_confidence_matches", [])
            if m["relationship"] == "duplicate"
        ]

        if duplicates:
            clusters[defect["id"]] = duplicates
            processed.update(duplicates)

    return clusters

Integrate at filing time, not triage time. The most effective duplicate detection happens when the bug is filed, not during triage. Add a pre-submission check: when a user starts typing a defect title, query the LLM to search for similar existing defects and display them. This prevents duplicates from entering the system at all — the reporter sees the existing bug and adds their information as a comment instead of creating a new ticket.

5. Severity and Priority Scoring

Severity (how bad is the bug?) and priority (how soon should we fix it?) are the two axes of defect triage. But teams routinely confuse them or apply them inconsistently. An LLM can apply a consistent scoring rubric across all defects.

SCORING_PROMPT = """You are a QA lead applying a standardized severity and
priority scoring rubric to defect reports. Score each defect using these
definitions:

SEVERITY (how bad is the impact):
- S1 Critical: System crash, data loss, security breach, complete feature failure
- S2 Major: Feature partially broken, significant user impact, no workaround
- S3 Moderate: Feature works but with issues, workaround available
- S4 Minor: Cosmetic issues, minor inconvenience, edge case

PRIORITY (how soon to fix):
- P1 Immediate: Fix now, block release if needed
- P2 High: Fix this sprint
- P3 Medium: Fix next sprint
- P4 Low: Fix when convenient, backlog

Consider these factors for priority:
1. Business impact (revenue, compliance, reputation)
2. User impact (how many users, how often)
3. Workaround availability
4. Fix complexity (quick fix vs. architectural change)
5. Regression risk (could the fix break other things?)

Return JSON with: severity, severity_rationale, priority,
priority_rationale, business_impact_score (1-10),
user_impact_score (1-10), fix_complexity (low/medium/high),
recommended_sprint (current/next/backlog)."""

def score_defect(defect: dict, business_context: str = "") -> dict:
    """Score a defect for severity and priority."""
    defect_text = json.dumps(defect, indent=2)
    context = f"\nBusiness context: {business_context}" if business_context else ""

    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": SCORING_PROMPT},
            {"role": "user", "content": defect_text + context}
        ],
        response_format={"type": "json_object"},
        temperature=0.2
    )
    return json.loads(response.choices[0].message.content)

print(f"SEVERITY: {score['severity']}")

print(f"  Rationale: {score['priority_rationale']}")

Calibrate with your team's historical decisions. Include 5-10 examples of previously triaged defects (with their final severity/priority) as few-shot examples in the prompt. This aligns the LLM's scoring with your team's actual standards, not just textbook definitions. Different teams legitimately have different thresholds — a cosmetic bug on a medical device UI might be S2 for one team and S4 for another.

6. Defect Pattern Recognition

Individual defects are symptoms. Patterns across defects reveal systemic problems. An LLM can analyze your defect backlog and identify recurring patterns that point to architectural issues, process failures, or skill gaps.

PATTERN_PROMPT = """You are a quality analytics specialist. Analyze a
collection of defect reports and identify systemic patterns.

Look for:
1. COMPONENT HOTSPOTS: Which modules have disproportionately many defects?
2. ROOT CAUSE CLUSTERS: What types of mistakes keep recurring?
3. TEMPORAL PATTERNS: Are defects clustered around certain events
   (releases, sprints, seasons)?
4. REGRESSION PATTERNS: Which areas keep breaking after fixes?
5. PROCESS GAPS: What do the defects reveal about gaps in requirements,
   code review, or testing?
6. SKILL GAPS: Do certain types of defects suggest training needs?

For each pattern, provide:
- pattern_name: descriptive name
- evidence: which defect IDs support this pattern
- frequency: how often it occurs
- impact: business impact of this pattern continuing
- recommendation: specific action to address the root cause
- effort: low/medium/high effort to fix

Return JSON with key "patterns" sorted by impact (highest first)."""

def find_defect_patterns(defects: list[dict]) -> list[dict]:
    """Analyze a defect collection for systemic patterns."""
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": PATTERN_PROMPT},
            {"role": "user", "content": f"""Analyze these {len(defects)} defects
for systemic patterns:

{json.dumps(defects, indent=2)}"""}
        ],
        response_format={"type": "json_object"},
        temperature=0.4
    )
    return json.loads(response.choices[0].message.content)["patterns"]

7. Regression Risk Assessment

Every bug fix carries regression risk — the fix might break something else. LLMs can assess this risk by analyzing the defect, the likely fix, and the system's dependency graph to predict what else might be affected.

REGRESSION_PROMPT = """You are a software architect assessing regression risk.
Given a defect and its probable fix, analyze:

1. DIRECTLY AFFECTED AREAS: What code/features will the fix touch?
2. INDIRECTLY AFFECTED AREAS: What depends on the changed code?
3. RISK LEVEL: high/medium/low for each affected area
4. RECOMMENDED REGRESSION TESTS: What existing tests to re-run?
5. NEW TESTS NEEDED: What new tests should be added to prevent regression?
6. DEPLOYMENT RISK: Should this be deployed alone or with other changes?
7. ROLLBACK PLAN: What to do if the fix causes regression in production?

Return JSON with these keys and an overall "regression_risk_score" (1-10)."""

def assess_regression_risk(defect: dict, proposed_fix: str,
                           system_context: str = "") -> dict:
    """Assess the regression risk of a proposed defect fix."""
    response = client.chat.completions.create(
        model="gpt-4o",
        messages=[
            {"role": "system", "content": REGRESSION_PROMPT},
            {"role": "user", "content": f"""Defect:
{json.dumps(defect, indent=2)}

Proposed Fix: {proposed_fix}

System Context: {system_context}"""}
        ],
        response_format={"type": "json_object"},
        temperature=0.3
    )
    return json.loads(response.choices[0].message.content)


# Assess regression risk for a payment fix
risk = assess_regression_risk(

High regression risk does not mean "don't fix it." It means "fix it carefully." A regression risk score of 7/10 tells the team to allocate extra testing time, deploy cautiously, and have a rollback plan ready. The worst outcome is fixing a bug quickly without considering regression, causing a production incident that is worse than the original bug.

Project: Defect Triage Assistant

Build an end-to-end defect triage assistant that processes incoming defect reports and produces a triage recommendation for each one.

Project Requirements

  1. Accept defect reports in JSON format (individual or batch)
  2. Classify each defect by type, component, and root cause
  3. Check for duplicates against existing backlog
  4. Score severity and priority using a configurable rubric
  5. Perform preliminary root cause analysis
  6. Generate a triage summary report
  7. Track triage decision accuracy over time

Starter Code

import json
from datetime import datetime
from pathlib import Path
from openai import OpenAI

client = OpenAI()

class DefectTriageAssistant:
    """Automated defect triage pipeline."""

    def __init__(self, backlog_path: str = "backlog.json",
                 model: str = "gpt-4o"):
        self.model = model
        self.backlog = self._load_backlog(backlog_path)
        self.triage_history = []

    def _load_backlog(self, path: str) -> list[dict]:
        """Load existing defect backlog."""
        if Path(path).exists():
            return json.loads(Path(path).read_text())
        return []

    def triage(self, defect: dict) -> dict:
        """Run full triage pipeline on a defect."""
        result = {
            "defect_id": defect["id"],
            "timestamp": datetime.now().isoformat(),
            "defect": defect,
        }

        # Step 1: Classification
        print(f"  Classifying {defect['id']}...")
        result["classification"] = classify_defect(defect)["classification"]

        # Step 2: Duplicate check
        print(f"  Checking for duplicates...")
        dup_result = detect_duplicates(defect, self.backlog)
        result["duplicate_check"] = {
            "verdict": dup_result["verdict"],
            "matches": dup_result.get("high_confidence_matches", [])
        }

        # Step 3: Severity/Priority scoring
        print(f"  Scoring severity and priority...")
        result["scoring"] = score_defect(defect)

        # Step 4: Root cause analysis
        print(f"  Analyzing root cause...")
        result["root_cause"] = analyze_root_cause(defect)

        # Step 5: Generate recommendation
        result["recommendation"] = self._generate_recommendation(result)

        self.triage_history.append(result)
        return result

    def _generate_recommendation(self, triage_result: dict) -> str:
        """Generate a human-readable triage recommendation."""
        is_duplicate = "duplicate" in triage_result["duplicate_check"]["verdict"].lower()
        severity = triage_result["scoring"]["severity"]
        priority = triage_result["scoring"]["priority"]

        if is_duplicate:
            match_id = triage_result["duplicate_check"]["matches"][0]["existing_id"]
            return (f"DUPLICATE — Close as duplicate of {match_id}. "
                    f"Add reporter's additional details as a comment.")

        return (f"TRIAGE: {severity} / {priority} — "
                f"Assign to {triage_result['classification'].get('suggested_assignee_team', 'unassigned')}. "
                f"Root cause: {triage_result['root_cause'].get('probable_root_cause', 'Unknown')}.")

    def triage_batch(self, defects: list[dict]) -> list[dict]:
        """Triage a batch of defects."""
        results = []
        for defect in defects:
            print(f"\nTriaging {defect['id']}: {defect['title']}")
            result = self.triage(defect)
            results.append(result)
            print(f"  => {result['recommendation']}")
        return results

    def generate_report(self, results: list[dict]) -> str:
        """Generate a triage summary report."""
        report_lines = [
            f"TRIAGE REPORT — {datetime.now().strftime('%Y-%m-%d %H:%M')}",
            f"{'='*60}",
            f"Defects triaged: {len(results)}",
            f"Duplicates found: {sum(1 for r in results if 'duplicate' in r['duplicate_check']['verdict'].lower())}",
            ""
        ]

        for r in results:
            report_lines.append(f"[{r['defect_id']}] {r['defect']['title']}")
            report_lines.append(f"  Severity: {r['scoring']['severity']} | "
                              f"Priority: {r['scoring']['priority']}")
            report_lines.append(f"  Recommendation: {r['recommendation']}")
            report_lines.append("")

        return "\n".join(report_lines)


# Usage
if __name__ == "__main__":
    assistant = DefectTriageAssistant("existing_backlog.json")

    new_defects = [
        {"id": "BUG-4590", "title": "Wire transfer over $10K gets stuck",
         "description": "..."},
        {"id": "BUG-4591", "title": "Dashboard chart renders blank on Safari",
         "description": "..."},
    ]

    results = assistant.triage_batch(new_defects)
    report = assistant.generate_report(results)
    print(report)

Extension Ideas

  • Integrate with Jira/GitHub Issues to pull defects automatically and push triage results back
  • Build a dashboard showing triage accuracy, common patterns, and backlog health over time

Summary

  • Automated classification provides consistent, multi-dimensional categorization of defects (type, component, root cause, reproducibility) in seconds.
  • Root cause analysis correlates defect descriptions with recent code changes and known issues to generate investigation hypotheses.
  • Duplicate detection uses semantic understanding to catch duplicates that keyword matching misses — reducing backlog bloat by 15-40%.
  • Severity and priority scoring applies a consistent rubric across all defects, reducing triage meeting time and improving decision quality.
  • Pattern recognition transforms individual defects into systemic insights — identifying component hotspots, recurring root causes, and process gaps.
  • Regression risk assessment predicts the blast radius of a fix, helping teams deploy safely.
  • Human judgment remains the final arbiter — LLM triage is a recommendation system, not a decision-making system.
